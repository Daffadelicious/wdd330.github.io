Testing and Debugging

'use strict'

'use strict' gives us more errors. It can help us to avoid using bad practice early on in our project so that even if that code would work there, it won't cause issues in the future. Strict is encouraged in basically every javascript project, there's no downside to having more help in debugging. IMPORTANT: 'use strict' MUST be the first line of a javascript file in order for it to work.

Linting Tools

Helps with syntax errors, indentation, and pointing out bad practice. Another huge benefit is it can act as a style guide, keeping a team working within the same conventions

Feature Detection

Essentially an if statment that checks to see if user is up to date or if API or other feature actually exists. Ensures we can give a custom error message or provide an alternative instead of giving the user a scary error.

Debugging in the Browser and Debuggers

My favorite, and my expertise! All the fun stuff, you've got alert() and console.log() which are the easiest in my opinion. Most proper browsers can debug now including Firefox, Edge, Chrome, and Safari. I also like using the debugger tool in VS code.

Error Objects

Great way to help correct errors on the user's part or to debug. You can declare specific messages for all sorts of errors (ex: EvalError, RangeError, SyntaxError, etc). It's always easier to solve an error if you know exactly what's causing it.

Throwing Exceptions

Basically an exit, great for shoving an important problem in our face so that the error can be solved before moving onto the next lines of code. Great way to debug line by line instead of stumbling across a giant tangled ball of mess later on that you'd have to pick apart.

Exception Handling

This is similar to throwing exceptions but code continue in the background and errors can be handed around in the background as opposed to in the user's face. This can allow for discrete fixes and keeps things moving for the user instead of having everything break over a very small or simple error. Not as great for initial debugging but better for once a site is out and functioning.

Testing and Frameworks

Self explanitory, test individual functions or snippets of codes to ensure they work before adding them to a huge project. Testing frameworks help us write tests, a popular one is Jest. There's a development style centered around this: TTD. The steps it follows are:

  1. Write tests (should initially fail)
  2. Write code to pass the tests
  3. Refactor the code
  4. Test refactured code
  5. Write more tests for new features
